End-to-End Architecture

From a Gemini prompt to a published page

Six stages, six different actors — the diagram below is the entire system.

You — outside the repo

Run the Gemini Gem

A fixed research prompt (GEMINI_PROMPT.md) is run by hand in the Gemini app. No API key, no code, no automation — entirely manual.

↓copy output

editor.html — your browser

Paste & publish

A single self-contained HTML admin tool. Parses the pasted Markdown/JSON (source: "gemini-app") into the story schema for Briefing or Flash, then calls the GitHub REST API directly.

3
GitHub repo — main branch

Data lands on main

data/<date>.json, data/flash-<date>.json, and data/index.json are committed straight to main — SHA-checked, no merge step.

↓push event triggers workflow
4
GitHub Actions — ssg.yml

Static pages get built

Runs npm run ssg (pipeline/ssg.js) — renders day/, story/, flash/, and OG share-image PNGs, then commits them back as "Auto-generate SSG assets [skip ci]".

↓git push (bot commit) → auto-deploy
5
Netlify

Site redeploys

Every push to main redeploys automatically — static files plus two serverless functions (subscribe.js, notify.js). netlify.toml adds an SPA-fallback redirect for the brief window before step 4 finishes.

↓HTTPS request
6
Reader's browser

Page renders

app.js (the SPA) takes over: serves the pre-rendered HTML when it exists, or boots client-side and fetches live JSON during the gap. Either way, the reader sees the story.

Zoom In — Stages 1 & 2

How content actually gets written

① The Gemini Gem

Lives entirely outside this repo. GEMINI_PROMPT.md is the prompt you paste into the Gemini app — it instructs Gemini to research that day's stories using web grounding and return a fixed Markdown structure: TL;DR, 5W1H, multi-perspective coverage, facts vs. claims, blind spots, editorial insight, impact analysis, context, and a simple explanation.

② editor.html — what it parses

Accepts either the raw Gemini Markdown or a hand-edited JSON payload, tags it source: "gemini-app", and converts it client-side into the story schema used by both products: Briefing (deep dives) and Flash (fast-scroll feed).

③ Publishing — no backend involved

editor.html talks directly to api.github.com using a personal access token you supply in the browser. It fetches the current file's SHA first (to avoid clobbering concurrent edits), then writes:

  • data/<date>.json — Briefing stories
  • data/flash-<date>.json — Flash items
  • data/index.json — refreshed list of available dates

④ Side path — push notifications

editor.html can also call /.netlify/functions/notify directly to send a web-push alert to subscribers when a briefing goes live. This is a separate Netlify Function — unrelated to the SSG/data pipeline above.

What happens if a reader hits the page mid-gap

Refreshing /day/<date> or sharing the link in WhatsApp/Twitter during the gap used to 404, or show a generic preview — Netlify had no static file to serve and no fallback rule.

The fix already shipped

netlify.toml adds a catch-all /* → /index.html (200) redirect. It only applies when no real file exists yet, so it masks exactly this gap — the SPA boots and renders from the already-published JSON instead of 404ing.

Zoom In — Stage 5

Two ways a request gets served

Path A — static page exists

Reader requests /story/<date>/<id>

→ Netlify finds the real story/<date>/<id>/index.html on disk

→ Served directly, full per-article og:image / og:title meta tags included

→ Crawlers (Twitter, WhatsApp, LinkedIn) see the correct share preview

Path B — mid-gap (page not built yet)

Reader requests the same URL

→ No static file matches → netlify.toml catch-all fires → serves index.html

→ SPA (app.js) boots, fetches data/<date>.json, renders client-side

→ Human readers see the story fine; crawlers still see the generic homepage preview until SSG lands

Also on Netlify

Two serverless functions, unrelated to the redirect rule: subscribe.js (stores push subscriptions in Netlify Blobs) and notify.js (sends the web-push). Both live under /.netlify/functions/, a reserved path the catch-all redirect never touches.

Data Model

Data is truth. Everything else is a cache.

data/*.jsonsource of truth — hand-published via editor.html
day/<date>/Briefing page
story/<date>/<id>/per-article + OG meta
flash/day/<date>/Flash page
og-images/share preview PNGs

Why this matters

Every box below the line is generated by pipeline/ssg.js and can always be rebuilt from the JSON above by re-running npm run ssg. None of it is ever hand-edited — if something looks wrong down there, the fix is upstream in the data, or in the build script itself.
